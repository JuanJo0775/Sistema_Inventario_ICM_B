# ADR-004: Implementar autenticacion con JWT (access + refresh tokens)

## Estado: Aceptado

## Contexto

El sistema expone una API REST consumida por un frontend separado (RNF-005). Se requiere autenticacion sin estado (stateless) para no mantener sesiones en servidor, compatible con la arquitectura de monolito modular. La restriccion horaria de los auxiliares (BR-03) exige que la validacion de acceso ocurra no solo en el login sino en cada request. Se necesita mecanismo de revocacion al deshabilitar usuarios (RF-002, Scenario 3) y al hacer logout. El equipo trabaja con Django REST Framework como framework de API.

## Decision

Se implementa autenticacion JWT con dos tokens diferenciados:

- **access_token**: vida corta (60 minutos), incluye user_id y role en payload, usado en cada request con Authorization: Bearer <token>.
- **refresh_token**: vida larga (7 dias), almacenado en cliente, usado para renovar el access token.

Al hacer logout o deshabilitar una cuenta, el refresh token se agrega a una blacklist para revocacion efectiva.

La validacion horaria (BR-03) se implementa como permiso DRF IsWithinOperatingHours que se evalua en cada request del auxiliar, no solo en el login.

Biblioteca: djangorestframework-simplejwt con soporte de blacklist habilitado.

## Alternativas descartadas

**Sesiones Django (cookies)**: Descartado porque requiere estado en servidor y complica el despliegue de multiples instancias futuras. No es compatible con el principio de separacion frontend/backend.

**API Keys estaticas**: Descartado porque no permite expiracion, restriccion horaria dinamica ni revocacion granular por usuario.

**OAuth2 completo**: Descartado por sobredimensionamiento; el sistema no tiene flujos de autorizacion de terceros y la complejidad de OAuth2 excede el alcance del proyecto academico.

## Consecuencias

### Positivas

- Autenticacion stateless: el servidor no mantiene estado de sesion, facilitando escalado horizontal futuro.
- El payload del JWT incluye el rol, permitiendo que IsAlmacenista, IsAuxiliarDespacho y IsWithinOperatingHours operen sin consultar BD en cada request.
- Revocacion real mediante blacklist al deshabilitar usuarios (RF-002 Scenario 3).
- Expiracion corta del access token (60 min) limita la ventana de exposicion si un token es comprometido.

### Negativas

- La blacklist de refresh tokens requiere consulta a BD en cada renovacion, introduciendo latencia minima en ese flujo.
- Si el role cambia en el perfil del usuario, los access tokens ya emitidos siguen siendo validos hasta su expiracion (hasta 60 min de desfase). Requiere considerar refresh forzado al modificar roles.
- La gestion de la blacklist crece indefinidamente si no se implementa limpieza periodica de tokens expirados.
- Los tokens JWT almacenados en el cliente (localStorage o similar) son vulnerables a XSS si el frontend no implementa las medidas de seguridad correspondientes.
