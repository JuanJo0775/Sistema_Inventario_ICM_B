# ADR-012: Alinear la imagen de produccion con dependencias runtime explicitas

## Estado: Aceptado

## Fecha: 2026-05-20

## Contexto

El stack de produccion arranca `gunicorn` desde `docker-compose.prod.yml`, pero `docker/Dockerfile` instala unicamente `requirements/base.txt`. Como `gunicorn` vive en `requirements/production.txt`, la imagen resultante no puede ejecutar el comando de produccion si no se incorpora ese archivo o una capa equivalente. Esto convierte la composicion de produccion en una restriccion operativa real: el runtime productivo no es equivalente al de desarrollo.

## Decision

Se documenta que cualquier despliegue productivo debe construir una imagen que incluya `requirements/production.txt` o una capa equivalente antes de invocar `gunicorn`. La imagen base de desarrollo no debe asumirse suficiente para produccion.

## Alternativas descartadas

**Usar `python manage.py runserver` en produccion**: descartado por no ser un servidor WSGI adecuado ni apto para trafico real.

**Instalar `gunicorn` manualmente en el contenedor en tiempo de ejecucion**: descartado porque vuelve el despliegue no reproducible y fragil.

**Mantener una unica imagen sin diferenciar dependencias**: descartado porque oculta el requisito explicito de runtime y dificulta auditar el despliegue.

## Consecuencias

### Positivas

- El requisito del servidor WSGI queda explicito.
- Se evita tratar la imagen de desarrollo como desplegable en produccion por accidente.
- El contrato entre `docker-compose.prod.yml`, `Dockerfile` y requirements queda auditable.

### Negativas

- El pipeline de build debe coordinar Dockerfile y requirements de produccion.
- Si cambia el servidor WSGI o sus dependencias, hay que actualizar la documentacion y el build.
- La imagen productiva requiere una comprobacion adicional de dependencias, no solo una build basica de Django.
