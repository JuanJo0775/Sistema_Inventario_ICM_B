# ADR-006: Exponer toda la funcionalidad mediante API REST versionada bajo /api/v1/

## Estado: Aceptado

## Contexto

El sistema tiene una arquitectura separada de frontend y backend (RNF-005). Multiples clientes pueden consumir el backend (frontend web, dispositivos moviles con lector HID, potencialmente integraciones contables futuras). El proyecto academico requiere documentacion de API como entregable verificable en el Corte 3 (RNF-005, Scenario 2). El sistema debe poder evolucionar sin romper clientes existentes, lo que requiere una estrategia de versionado desde el inicio.

## Decision

Se implementa una API REST bajo el prefijo /api/v1/ con los siguientes dominios: /auth/, /catalog/, /inventory/, /movements/, /reports/, /alerts/, /audit/. Todos los contratos siguen el formato application/json con estructura uniforme de errores (error, message, detail). La documentacion se genera automaticamente con drf-spectacular y se expone en /api/v1/docs/. Los cambios incompatibles requieren crear /api/v2/ en lugar de modificar endpoints existentes.

## Alternativas descartadas

**GraphQL**: Descartado porque la curva de aprendizaje (esquemas, resolvers, N+1 en GraphQL) supera el beneficio para un equipo academico con plazo acotado. Los clientes del sistema son conocidos y los contratos de datos son estables.

**API sin versionado**: Descartado porque cualquier cambio de contrato romperia el frontend sin una via de migracion controlada.

**RPC / gRPC**: Descartado por incompatibilidad con el patron de consumo desde navegador y por overhead de setup innecesario para el alcance del proyecto.

## Consecuencias

### Positivas

- Desacoplamiento total frontend/backend: cada capa puede evolucionar independientemente.
- El versionado /api/v1/ permite introducir /api/v2/ para cambios incompatibles sin romper clientes existentes.
- Swagger autogenerado por drf-spectacular cumple el entregable de documentacion del Corte 3 sin esfuerzo manual adicional.
- Facilita pruebas de integracion automatizadas y contratos verificables.
- Permite que el sistema sea consumido por integraciones externas futuras (software contable de ICM) sin modificar el backend.

### Negativas

- REST sobre HTTP agrega latencia de red en cada operacion; irrelevante para el volumen actual pero a considerar si se aaden flujos en tiempo real.
- Sin un cliente de tipos generados (TypeScript, etc.), los contratos de API no se validan en tiempo de compilacion en el frontend, dejando errores de contrato para tiempo de ejecucion.
- drf-spectacular requiere que los desarrolladores mantengan actualizadas las anotaciones en los ViewSets; si se omiten, la documentacion puede quedar desincronizada.
