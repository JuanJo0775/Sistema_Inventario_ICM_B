# ADR-007: Implementar control de acceso basado en roles (RBAC) con permisos DRF componibles

## Estado: Aceptado

## Contexto

El sistema tiene tres roles con permisos radicalmente diferentes (almacenista, auxiliar_despacho, administrador) y una regla de restriccion horaria que aplica solo a uno de ellos (BR-03). Existen operaciones que requieren combinacion de condiciones: por ejemplo, un despacho requiere estar autenticado, tener rol auxiliar y estar dentro del horario. Las reglas BR-01 a BR-03 son de obligatorio cumplimiento y deben verificarse en cada request, no solo en el login. El framework base es Django REST Framework.

## Decision

Se implementa RBAC mediante permisos DRF granulares y componibles definidos en shared/permissions.py y apps/*/permissions.py:

- **IsAlmacenista, IsAuxiliarDespacho, IsAdministrador**: verifican request.user.role.
- **IsWithinOperatingHours**: verifica franja horaria en timezone America/Bogota para auxiliares.

Se combinan en cada ViewSet mediante permission_classes = [IsAuthenticated, IsAlmacenista, IsWithinOperatingHours].

El campo role del usuario se almacena en el modelo User (extension de AbstractUser) y en el payload del JWT para evitar consulta a BD en cada validacion.

## Alternativas descartadas

**Django's built-in Groups/Permissions**: Descartado porque el modelo de permisos de Django esta orientado a permisos por objeto/modelo, no a roles de negocio con restricciones contextuales (como la franja horaria). Requeriria workarounds complejos para BR-03.

**Middleware centralizado de autorizacion**: Descartado porque un middleware unico que evalua todos los endpoints seria dificil de mantener y probar; la granularidad por ViewSet con permisos componibles es mas mantenible y explicita.

**Autorizacion basada en claims/scopes JWT solamente**: Descartado porque no permite logica dinamica como la verificacion de franja horaria en tiempo de request.

## Consecuencias

### Positivas

- Cada endpoint declara explicitamente sus requisitos de autorizacion, facilitando auditoria de seguridad.
- Los permisos son testeables de forma aislada sin levantar el flujo completo de negocio.
- La componibilidad permite reutilizar IsWithinOperatingHours en todos los endpoints del auxiliar sin duplicar codigo.
- Cumplimiento directo de BR-01, BR-02 y BR-03 a nivel de infraestructura, no solo de aplicacion.

### Negativas

- La verificacion de IsWithinOperatingHours en cada request del auxiliar tiene un costo minimo de computo (lectura del reloj + comparacion), aunque despreciable.
- Si se agrega un nuevo endpoint y el desarrollador omite declarar permission_classes, el endpoint podria quedar sin proteccion. No hay mecanismo que fuerce la declaracion explicita de permisos.
- El rol esta embebido en el JWT: un cambio de rol no se refleja hasta que el token actual expire (hasta 60 min), a menos que se implemente un mecanismo de invalidacion activa.
