# ADR-002: Implementar separacion estricta de responsabilidades en capas

## Estado: Aceptado

## Contexto

El proyecto involucra tres asignaturas simultaneas (Arquitectura de Software I, Programacion Web, Pruebas de Software) y estudiantes de Ingenieria Industrial como clientes funcionales. El codigo sera evaluado formalmente en el Corte 3 mediante una revision arquitectonica que verifica el cumplimiento de principios SOLID (RNF-005). La experiencia en proyectos academicos similares muestra que la logica de negocio tiende a filtrarse hacia vistas y modelos cuando no hay una convencion explicita, generando codigo dificil de probar y de mantener. Ademas, el sistema tiene 13 reglas de negocio (BR-01 a BR-13) con validaciones cruzadas complejas que deben ser testeables de forma aislada.

## Decision

Se establece un modelo de capas explicito y obligatorio dentro de cada app Django:

- **models.py**: solo estructura de datos y constraints de BD. Sin logica de negocio.
- **serializers.py**: solo validacion de formato y transformacion entrada/salida.
- **views.py**: solo adaptador HTTP; orquesta recepcion y delega a servicios.
- **services.py**: toda la logica de negocio y reglas de dominio (BR-XX).
- **selectors.py**: consultas de lectura complejas sin efectos secundarios (patron Query Object).
- **permissions.py**: reglas de autorizacion DRF.

Esta separacion implementa CQRS-lite: services.py para comandos (escritura) y selectors.py para consultas (lectura).

## Alternativas descartadas

**Fat models (logica en models.py)**: Descartado porque acopla la logica de negocio al ORM, impide reutilizacion desde CLI o tareas asincronas, y dificulta las pruebas unitarias al requerir acceso a base de datos.

**Fat views (logica en views.py)**: Descartado porque mezcla protocolo HTTP con reglas de dominio, impide reutilizar la logica desde otros contextos (tests, comandos de gestion) y viola SRP.

**Serializer-driven logic**: Descartado porque las validaciones de negocio (ej. BR-08 validacion cruzada, BR-04 serial obligatorio) requieren acceso a multiples entidades y contexto transaccional que no pertenece a la capa de serializacion.

## Consecuencias

### Positivas

- Las 13 reglas de negocio son testeables en services.py sin levantar servidor HTTP.
- Las vistas permanecen delgadas, facilitando el cambio de framework de presentacion sin tocar logica de dominio.
- selectors.py permite optimizar consultas de reporte (RNF-004) de forma independiente.
- Cumplimiento directo y verificable de SRP, OCP y DIP de SOLID (RNF-005).
- Facilita la incorporacion de nuevos desarrolladores al proyecto: la convencion es explicita y no ambigua.

### Negativas

- Mayor numero de archivos por app aumenta la friccion inicial para desarrolladores acostumbrados a proyectos Django convencionales.
- El principio "logica NUNCA en models/serializers/views" requiere disciplina constante y revision de codigo activa; no hay mecanismo tecnico que lo imponga automaticamente.
- La separacion services/selectors puede generar duplicacion leve cuando una operacion de escritura necesita leer datos complejos para validar.
