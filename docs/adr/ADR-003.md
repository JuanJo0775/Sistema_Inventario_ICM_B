# ADR-003: Usar PostgreSQL como base de datos relacional unica del sistema

## Estado: Aceptado

## Contexto

El sistema requiere consistencia transaccional fuerte para el modulo de inventario (BR-11): cada movimiento debe actualizar el stock derivado en la misma transaccion atomica, con soporte para select_for_update() para prevenir race conditions cuando multiples auxiliares despachan simultaneamente. Se necesita soporte para constraints de integridad referencial (FK entre Movement, Product, Location, User), constraint CHECK (current_stock >= 0) para impedir stock negativo, e indices compuestos sobre la tabla Movement para consultas de reportes eficientes (RNF-004). La inmutabilidad del ledger (BR-10) requiere una BD con soporte robusto para transacciones ACID. El equipo trabaja con Django ORM y el stack academico disponible.

## Decision

Se adopta PostgreSQL 18 como unica base de datos del sistema, desplegada como servicio Docker (postgres:18-alpine). Se configura con usuario/contrasena dedicados (icm_user/icm_db) y volumen persistente. Django ORM gestiona toda la comunicacion con la BD.

## Alternativas descartadas

**SQLite**: Descartado porque no soporta select_for_update() requerido para prevenir race conditions en despacho concurrente (BR-08, BR-11), y no esta disenado para multiples escrituras concurrentes en produccion.

**MySQL/MariaDB**: Descartado por menor integracion con el ecosistema Django/DRF en terminos de tipo de dato (JSONField nativo, UUID primario), y menor soporte para constraints CHECK en versiones antiguas.

**MongoDB o BD NoSQL**: Descartado porque el modelo de datos es altamente relacional (movimientos -> productos -> ubicaciones -> usuarios) y las garantias ACID son no negociables para el ledger de inventario.

## Consecuencias

### Positivas

- Soporte nativo para transacciones ACID y SELECT FOR UPDATE, garantizando BR-11 bajo concurrencia.
- Constraints de integridad referencial y CHECK a nivel de BD como ultima linea de defensa.
- JSONField nativo para customer_data en Movement sin extensiones adicionales.
- Soporte para UUID como clave primaria sin overhead.
- Ecosistema maduro de herramientas de monitoreo y backup.

### Negativas

- Requiere servicio Docker adicional y configuracion de volumen persistente, aumentando la complejidad del entorno de desarrollo frente a SQLite.
- Las migraciones de Django deben ejecutarse correctamente en cada despliegue; un fallo de migracion en produccion bloquea el arranque del sistema.
- Sin configuracion de replicas de lectura (no planificada en este alcance), todas las consultas de reportes compiten con las escrituras transaccionales en el mismo servidor.
