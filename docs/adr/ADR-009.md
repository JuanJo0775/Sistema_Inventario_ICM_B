# ADR-009: Separar configuraciones de settings en base/development/production

## Estado: Aceptado

## Contexto

El sistema tiene comportamientos radicalmente diferentes segun el entorno: en desarrollo se necesita DEBUG=True, CORS abierto y email en consola; en produccion se requiere DEBUG=False, HTTPS forzado, headers de seguridad y CORS restringido. RNF-003 exige que ningun secreto (claves JWT, contrasenas de BD) sea almacenado en el codigo fuente. El proyecto usa un repositorio Git compartido entre el equipo academico, lo que hace critico impedir que credenciales reales lleguen al repositorio.

## Decision

Se adopta el patron de settings jerarquicos: config/settings/base.py (configuracion compartida), config/settings/development.py (sobreescrituras de desarrollo) y config/settings/production.py (sobreescrituras de produccion). El settings activo se selecciona mediante la variable de entorno DJANGO_SETTINGS_MODULE. Todos los secretos y configuraciones sensibles (SECRET_KEY, contrasenas de BD, configuracion JWT) se leen desde variables de entorno usando os.environ o django-environ. El archivo .env real esta en .gitignore; el repositorio solo incluye .env.example con valores de plantilla.

## Alternativas descartadas

**Unico settings.py con condicionales if DEBUG**: Descartado porque mezcla configuraciones de entornos diferentes en un solo archivo, dificultando la revision y aumentando el riesgo de activar configuraciones de desarrollo en produccion por error.

**Hardcodear secretos en settings**: Descartado de forma absoluta porque viola RNF-003 y expone credenciales en el repositorio Git, comprometiendo la seguridad del sistema.

**Gestion de secretos con AWS Secrets Manager o Vault**: Descartado por sobredimensionamiento para el alcance academico del proyecto.

## Consecuencias

### Positivas

- Ningun secreto en el repositorio; cumple RNF-003 desde el diseno.
- Los entornos son reproducibles y predecibles: no hay riesgo de activar DEBUG en produccion por olvido.
- .env.example sirve como documentacion viva de todas las variables que el sistema requiere.
- Facilita el despliegue en CI/CD (GitHub Actions) inyectando variables de entorno sin modificar codigo.

### Negativas

- Si un desarrollador no copia .env.example a .env al hacer checkout del repositorio, el sistema no arranca sin mensajes de error claros. Requiere documentacion explicita en el README.
- La proliferacion de variables de entorno sin validacion al arranque puede generar errores silenciosos si una variable critica esta mal configurada o ausente.
