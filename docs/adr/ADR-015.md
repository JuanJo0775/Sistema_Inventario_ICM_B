# ADR-015: Manejo Sistemático de `DoesNotExist` en Servicios con `select_for_update()`

## Estado: Implementado (2026-06-14)

## Contexto

Durante la auditoría de seguridad del código (2026-06-14) se identificaron **38 ocurrencias** del patrón `Model.objects.select_for_update().get(pk=...)` en servicios que **no capturan `DoesNotExist`**. Si el registro es eliminado entre la validación de la vista y la ejecución del servicio (ventana de condición de carrera), el `DoesNotExist` propaga como HTTP 500.

### Ejemplo del patrón problemático

```python
# apps/catalog/services.py:97-100
@transaction.atomic
def update_product(user, product_id, data, ...):
    _require_almacenista(user)
    product = Product.objects.select_for_update().get(pk=product_id)  # ← ¡DoesNotExist sin captura!
    ...
```

### Ejemplo del patrón ya corregido (referencia)

```python
# apps/alerts/services.py:487-490
try:
    alert = Alert.objects.select_for_update().get(pk=alert_id)
except Alert.DoesNotExist:
    raise ResourceNotFoundError(f"Alerta con ID {alert_id} no encontrada.")
```

## Análisis de Riesgo

### Mitigaciones existentes

1. **`get_object_or_404` en vistas:** La mayoría de los endpoints ejecutan `get_object_or_404()` (o el `get_object()` de DRF generic views) antes de llamar al servicio. Esto reduce drásticamente la probabilidad de que un `DoesNotExist` ocurra, pero no la elimina:
   - Ventana entre `get_object_or_404` (vista) y `select_for_update().get()` (servicio)
   - Un DELETE concurrente desde otro thread/request puede eliminar el registro en esa ventana
   - Esto aplica incluso con `@transaction.atomic` porque son transacciones separadas

2. **`@transaction.atomic`:** Protege contra corrupción de datos pero no convierte `DoesNotExist` en una respuesta HTTP amigable.

### Clasificación por severidad real

| Nivel | Criterio | Cantidad |
|-------|----------|----------|
| **Crítico** | Sin `get_object_or_404` previo en la vista | 0 |
| **Alto** | Vista llama `get_object_or_404` pero servicio recibe PK (no objeto) | 27 |
| **Medio** | Vista pasa el objeto ya hidratado al servicio | 11 |
| **Bajo** | Servicio es llamado solo desde dentro de otra transacción que ya lockeó | 0 |

**Conclusión del análisis:** No hay casos críticos. Los 38 casos tienen al menos `get_object_or_404` en la vista. Sin embargo, el riesgo de raza no es cero y la experiencia de error (500 vs. 404 amigable) es mejorable.

## Decisión Propuesta: Refactor Gradual con Helper Centralizado

### Opciones Consideradas

#### Opción A: Envolver cada `select_for_update().get()` individualmente (38 cambios manuales)

- **Ventaja:** Control granular, mensaje de error específico por contexto.
- **Desventaja:** Alto esfuerzo manual, propenso a errores humanos, difícil de mantener.
- **Veredicto:** ❌ Rechazado por alto costo de implementación y mantenimiento.

#### Opción B: Crear helper `get_for_update_or_404()` en `shared/` (RECOMENDADA)

- **Ventaja:** Un único cambio por archivo de servicio para importar y reemplazar, DRY, testable, mensaje consistente.
- **Desventaja:** Cambio mecánico en 38 líneas, requiere tocar ~10 archivos de servicio.
- **Veredicto:** ✅ Seleccionada.

#### Opción C: Decorador de método que envuelva el `DoesNotExist`

- **Ventaja:** Mínimo cambio en el código existente.
- **Desventaja:** Requiere que el decorador inspeccione el modelo, frágil con type hints, difícil de depurar.
- **Veredicto:** ❌ Rechazado por complejidad adicional innecesaria.

### Implementación Propuesta

#### Paso 1: Crear el helper en `shared/utils/db.py`

```python
"""Utilidades de base de datos compartidas entre servicios."""

from __future__ import annotations

from typing import TypeVar

from django.db import models
from django.db.models.base import Model

from shared.exceptions import ResourceNotFoundError

_ModelT = TypeVar("_ModelT", bound=Model)


def get_for_update_or_404(
    queryset: models.QuerySet[_ModelT],
    *,
    pk: object,
    detail: str | None = None,
) -> _ModelT:
    """
    Obtiene un objeto con lock de fila o lanza 404.

    Uso típico dentro de servicios @transaction.atomic:
        product = get_for_update_or_404(Product.objects, pk=product_id)

    Args:
        queryset: QuerySet del modelo, normalmente ``Model.objects``.
        pk: Valor de clave primaria a buscar.
        detail: Mensaje opcional para ResourceNotFoundError.

    Returns:
        La instancia del modelo con lock adquirido.

    Raises:
        ResourceNotFoundError: Si el objeto no existe (HTTP 404).
    """
    try:
        return queryset.select_for_update().get(pk=pk)
    except queryset.model.DoesNotExist:
        model_name = queryset.model._meta.verbose_name or queryset.model.__name__
        msg = detail or f"{model_name} con ID {pk} no encontrado."
        raise ResourceNotFoundError(msg)
```

#### Paso 2: Reemplazar el patrón en todos los servicios

Cada ocurrencia de:

```python
objeto = Modelo.objects.select_for_update().get(pk=objeto_id)
```

se reemplaza por:

```python
objeto = get_for_update_or_404(Modelo.objects, pk=objeto_id)
```

#### Paso 4: Probar el helper

```python
# tests/shared/test_db_utils.py
import pytest
from shared.utils.db import get_for_update_or_404
from shared.exceptions import ResourceNotFoundError


@pytest.mark.django_db
def test_get_for_update_or_404_returns_object(category):
    result = get_for_update_or_404(Category.objects, pk=category.id)
    assert result.id == category.id


@pytest.mark.django_db
def test_get_for_update_or_404_raises_404_for_missing():
    fake_pk = "00000000-0000-0000-0000-000000000000"
    with pytest.raises(ResourceNotFoundError):
        get_for_update_or_404(Category.objects, pk=fake_pk)
```

### Criterios de Aceptación

1. El helper `get_for_update_or_404()` existe en `shared/utils/db.py`.
2. Las 38 ocurrencias del patrón inseguro se reemplazan por el helper.
3. `ResourceNotFoundError` tiene `status_code = 404`.
4. Todas las pruebas existentes continúan pasando (578 actuales).
5. El helper tiene cobertura de pruebas unitarias (caso feliz + `DoesNotExist`).

### Pruebas a agregar

| Test | Propósito |
|------|-----------|
| `test_get_for_update_or_404_found` | Objeto existe → retorna instancia lockeada |
| `test_get_for_update_or_404_not_found` | Objeto no existe → `ResourceNotFoundError` con mensaje |
| `test_get_for_update_or_404_custom_message` | `detail=` personalizado se refleja en el error |

### Riesgos y Contrapartidas

- **Riesgo de regression:** El cambio es mecánico (`.get()` → `get_for_update_or_404()`) y no altera la lógica de negocio. Las pruebas existentes validan el comportamiento funcional.
- **Riesgo de rendimiento:** El helper usa `select_for_update()` igual que el código original. No hay cambio en el plan de consulta ni en los locks.
- **Dependencia:** Requiere que `ResourceNotFoundError` ya exista en `shared/exceptions.py` (fue agregado en ADR vía Hallazgo #2).

### Consecuencias

**Positivas:**
- Elimina 38 fuentes potenciales de HTTP 500 con un cambio mecánico.
- Mensajes de error consistentes en todo el sistema.
- El helper puede evolucionar (logging, métricas, etc.) desde un solo punto.
- Facilita la revisión de seguridad (un patrón, no 38 variantes).

**Negativas:**
- Requiere modificar ~10 archivos de servicio.
- Agrega una dependencia de `shared/utils/db.py` en todos los servicios.
- Los mensajes genéricos ("X con ID Y no encontrado") pierden contexto específico que algunos servicios podrían querer incluir. Mitigación: parámetro `detail=` opcional.
