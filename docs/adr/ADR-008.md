# ADR-008: Contenerizar el sistema completo con Docker Compose

## Estado: Aceptado

## Contexto

El proyecto es desarrollado por multiples estudiantes en equipos con diferentes sistemas operativos (Windows, macOS, Linux) y configuraciones locales heterogeneas. Los entregables academicos incluyen despliegue funcional en el Corte 3. Sin paridad de entornos, los errores del tipo "funciona en mi maquina" consumen tiempo de debugging que el calendario academico no permite. El sistema requiere PostgreSQL 15 con configuracion especifica, y las migraciones deben ejecutarse antes de que el servidor HTTP acepte trafico.

## Decision

Se adopta Docker Compose como mecanismo de orquestacion local y de despliegue del MVP, con dos servicios: db (PostgreSQL 15-alpine) y web (Django). El entrypoint.sh garantiza el orden de inicializacion: esperar a que PostgreSQL este listo (health check pg_isready), ejecutar migraciones, recopilar statics y arrancar el servidor. Los secretos se gestionan mediante archivo .env (local) o variables de entorno (produccion), nunca hardcodeados. Se mantienen configuraciones separadas para development (docker-compose.yml) y production (docker-compose.prod.yml).

## Alternativas descartadas

**Instalacion manual de dependencias por desarrollador**: Descartado porque genera entornos divergentes, especialmente en la version de PostgreSQL y las variables de entorno, causando errores intermitentes dificiles de reproducir.

**Kubernetes**: Descartado por sobredimensionamiento total para el alcance del proyecto; la complejidad operacional (pods, services, ingress, secrets) es incompatible con el plazo academico.

**Entorno virtual Python sin Docker para BD**: Descartado porque no garantiza la version ni la configuracion de PostgreSQL, y la gestion de health checks del arranque quedaria sin resolver.

## Consecuencias

### Positivas

- Paridad garantizada entre el entorno de todos los desarrolladores y el entorno de despliegue.
- El entrypoint.sh elimina la clase de bugs "el servidor arranco antes de que la BD estuviera lista".
- Onboarding de nuevos desarrolladores reducido a docker compose up.
- Las variables de entorno en .env.example documentan todas las configuraciones requeridas.
- Facilita la integracion continua en GitHub Actions con el servicio PostgreSQL como contenedor de testing.

### Negativas

- Docker aade overhead de recursos en maquinas con poca RAM; puede impactar el rendimiento en equipos de desarrollo con especificaciones bajas.
- Los estudiantes sin experiencia previa con Docker necesitan una curva de aprendizaje inicial (aunque corta) antes de poder contribuir.
- El docker-compose.yml actual no configura replicas ni load balancing; escalar horizontalmente requeriria migrar a una herramienta de orquestacion mas robusta.
- Los archivos del volumen de PostgreSQL en el host local pueden generar conflictos si se cambia la version de la imagen sin limpiar el volumen previo.
